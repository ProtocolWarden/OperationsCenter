## Overview

Recent fixes to board_worker (task re-promotion loops) and board_unblock (stale task detection, false-positive filtering) have stabilized the queue. This campaign adds durable health monitoring and resilience patterns to detect subtle queue degradation before it impacts task throughput. Four focused improvements: queue state consistency audits, task lifecycle trace instrumentation, label-validation hardening, and memory-pressure responsiveness.

## Goals

1. **Queue state consistency detector** — New `audit_queue_consistency()` function in board_unblock that detects orphaned tasks (Running/Blocked indefinitely, contradictory label states, missing parent dependencies). Run post-rule-application to emit a structured audit report with counts and recovery recommendations. Target: detect 3+ classes of orphaned patterns; false-positive rate <1%.

2. **Task lifecycle trace instrumentation** — Add structured logging at every state transition in board_worker (claim → plan → execute → post-execute transitions) and board_unblock rule applications. Log decision rationale, rule name, previous/new state, and affected label changes. Enable operators to reconstruct timeline of any task's lifecycle for debugging. Target: <5ms per decision log.

3. **Label validation and normalization** — Consolidate label-parsing helpers across board_worker and board_unblock with explicit whitespace/case handling, rate-limit spammy logs, and fail safely on malformed labels. Add a `LabelValidator` with tests for edge cases (whitespace, truncation, missing colons, duplicate prefixes). Target: 100% consistency on label extraction across both modules.

4. **Memory-pressure queue throttling** — When system memory drops below `_MEM_SKIP_THRESHOLD_GB`, queue high-memory tasks (spec-author, improve campaigns) to Backlog and reduce board_worker poll frequency to 5m (from 1m). Add exponential backoff recovery when memory recovers. Target: prevent executor OOM kills on memory-starved systems; smooth recovery when pressure releases.

## Constraints

- **No breaking changes** — board_worker and board_unblock behavior unchanged; improvements are additive (logging, audit output, label normalization).
- **No new dependencies** — Use stdlib logging, pathlib, and existing PlaneClient only.
- **Testing patterns** — Unit tests mock PlaneClient; integration tests use FakePlaneClient. No live Plane access required.
- **Backward-compatible transitions** — Existing tasks in any queue state must remain processable; no migrations required.
- **Observability first** — All decision points emit structured logs; enable future automation (alerts, dashboards) without code changes.

## Success Criteria

- **Audit function** — `audit_queue_consistency()` added to board_unblock; detects ≥3 orphan patterns; audit report includes task count, state distribution, recovery steps per pattern.
- **Trace logs** — Every board_worker transition (claim/plan/execute/post) and board_unblock rule application logs: `task_id`, `role`, `previous_state`, `new_state`, `decision_rationale`, `rule_name` (for unblock).
- **Label validator** — New `LabelValidator` class with `parse_label_value()`, `has_label()`, `has_label_prefix()`, `normalize_label()` methods; 100% test coverage; used by both modules.
- **Memory throttling** — board_worker respects `_MEM_SKIP_THRESHOLD_GB`; poll interval increases from 1m → 5m when memory < threshold; exponential recovery when memory > threshold + margin (10GB).
- **No regressions** — Full test suite passes; no change in task throughput on memory-healthy systems.
