## Overview

Recent work on per-model worker-backend cooldowns (including stale cooldown clearing in #218) provides robust timeout recovery, but operators lack comprehensive visibility into cooldown state transitions and the impact on executor fallback behavior. This campaign hardens cooldown state testing, adds structured observability metrics, and ensures model-aware fallback paths are exercised under realistic cooldown conditions.

## Goals

1. **Worker-backend cooldown state transition tests** — Add integration tests for cooldown detection, clearing, and recovery across ExecutionCoordinator and executor adapters. Cover normal timeout, stale-cooldown probe-and-clear, model-specific cooldown gates, and fallback model selection. Verify at least 4 distinct cooldown scenarios (timeout → recovery, stale clear, cascade fallback, backend recovery) with ≥80% line coverage for cooldown-related code paths.

2. **Cooldown state metrics and structured logging** — Implement JSON-structured cooldown event logging including `{ event_type, backend, model, timestamp_utc, reason, prior_state, new_state }`. Emit metrics for cooldown transitions (onset, clear, stale-probe). Add operator-facing logs for fallback activation. Success: cooldown state changes are queryable and ops can correlate with executor behavior.

3. **Model-aware fallback testing under cooldown** — Add tests verifying that when a backend hits cooldown, executor correctly selects fallback model (opus for sonnet per recent changes) and that retry logic respects both primary and fallback cooldown gates. Verify cascading fallback behavior when multiple models are saturated. Coverage: 3–4 scenarios with realistic timeout values.

4. **Cooldown recovery observability for operators** — Document cooldown detection procedure (probe vs. timeout detection) and recovery timeline expectations. Add operator runbook for diagnosing stuck cooldowns and manual recovery steps. Include example queries for extracting cooldown state from logs. Success: ops can identify and recover from stale cooldowns without code inspection.

## Constraints

- **Scope:** Testing, instrumentation, and operator documentation only — no changes to cooldown timeout calculation, gate logic, or backend selection policy.
- **Allowed paths:** Add tests under `tests/executor/cooldowns/` and `tests/executor/adapters/`; add structured logging in `src/operations_center/executor/cooldown_manager.py` and adapter dispatch paths; add operator runbook to `docs/operator_guides/`.
- **Avoid:** Changes to cooldown threshold values, backend priority ranking, or gate enforcement rules; refactoring of cooldown state machine without concrete test failure first.
- **Model-aware fallback testing:** Must exercise the new opus-fallback logic for sonnet; defer other model combinations to future work.
- **Measurement overhead:** Cooldown state tracking must add <5ms per executor dispatch; measure overhead before declaring success.
