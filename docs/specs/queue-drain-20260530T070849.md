## Overview

The `spec_hygiene` and `spec_trigger` modules were extracted from the legacy `spec_director` entrypoint as part of ADR 0007 maintenance task refactoring and are now production-critical for the spec-authoring pipeline. These modules coordinate all spec-driven campaigns but currently rely on smoke tests and lack comprehensive unit test coverage for edge cases, error recovery, and state transitions. This campaign hardens test coverage to prevent regressions in the campaign lifecycle and trigger detection logic.

## Goals

1. **Unit test spec_trigger trigger detection and deduplication** — Add focused tests for `TriggerDetection`, `plan_from_trigger()`, and deduplication logic covering valid triggers, missing fields, state-dependent filtering, and duplicate prevention. Target ≥80% line coverage for the module.

2. **Unit test spec_hygiene campaign lifecycle orchestration** — Add tests for campaign state transitions, phase detection logic, auto-promotion logic, and spec parsing. Cover normal paths and error cases (missing specs, corrupt YAML, state conflicts). Target ≥80% line coverage.

3. **Error recovery and concurrent state handling** — Add tests for error paths: missing records, concurrent updates, transient I/O failures, and recovery behavior. Verify both modules fail safely and log appropriately without corrupting downstream state.

## Constraints

- **Scope:** Unit tests only — no refactoring of the modules themselves unless required to make tests cleaner.
- **Allowed paths:** Use existing test infrastructure (pytest, mocks, fixtures in the test suite). Prioritize isolated unit tests over integration tests.
- **Avoid:** Changing public APIs or module behavior to make testing easier; tests should validate existing behavior.
- **Target modules:** `src/operations_center/spec_hygiene/` and `src/operations_center/spec_trigger/` only.
- **Preserve existing:** Do not modify or remove existing smoke test coverage; add focused unit tests alongside.

## Success Criteria
