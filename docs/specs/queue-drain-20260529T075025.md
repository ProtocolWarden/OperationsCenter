## Overview

The queue-drain subsystem (QueueHealingEngine + board-unblock rules) prevents task deadlock by automatically resolving stuck patterns. Recent fixes have addressed spec_trigger floods and thin-goal re-promotion loops. This campaign hardens the rate-gating logic, adds comprehensive test coverage for the flooding prevention mechanism, and ensures the 8 board-unblock rules execute correctly under concurrent task states.

## Goals

1. **Add unit tests for QueueHealingEngine decision logic** — Create `tests/unit/queue_healing/test_engine.py` with ≥12 tests covering: stale-blocked detection, duplicate-suppression deadlock resolution, recovery attempt budget exhaustion, retry-safe state transitions, and boundary cases around the staleness threshold. Verify that staleness calculation handles timezone-aware/naive datetimes correctly.

2. **Extend board-unblock rule tests with rate-gating edge cases** — Add 8 tests to `tests/unit/entrypoints/maintenance/test_board_unblock.py` covering: (a) Rules 7 & 8 newly added in recent commits, (b) concurrent state mutations (task moves from Blocked → Ready between rule evaluations), (c) thin-goal filtering in GOAL_BACKLOG_PROMOTE rule, (d) memory-threshold bypass conditions during low-memory states. Each test must verify the rule fires or skips as expected.

3. **Implement rate-gating metrics and audit trail** — Add a `QueueDrainMetrics` dataclass and emit decision counts per transition type (none / blocked_to_backlog / blocked_to_ready_for_ai / escalate / stale_lock_cleanup) to a JSON artifact. Verify that on high-volume drains (50+ decisions per cycle), the audit trail captures all transitions and prevents spec_trigger re-firing within a 5-minute window.

4. **Verify board-unblock handles Plane API errors gracefully** — Add ≥4 tests covering transient Plane failures during task state transitions (connection timeout, 429 rate-limit, task-not-found 404, malformed-response 500). Verify that: (a) individual rule failures do not crash the full unblock run, (b) failures are logged with task IDs for operator triage, (c) a dry-run mode (--dry-run flag) succeeds even if mutations would fail.

## Constraints

- All new tests must be pure unit tests using mocks; no live Plane API calls or Docker.
- Do not modify the QueueHealingEngine or board_unblock logic itself — only test coverage.
- Metrics emission must be optional (off by default) and not block rule application.
- Dry-run must not mutate Plane state; use `--dry-run` flag to gate all `client.update_task()` calls.
- Keep each test file ≤300 lines to fit in a single executor run.
- The staleness threshold must remain configurable (default 3600s for QueueHealingEngine, 4h for board-unblock).

## Success Criteria

- `pytest tests/unit/queue_healing/test_engine.py` passes with ≥12 tests, all green.
