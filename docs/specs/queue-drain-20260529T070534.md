## Overview

Queue-drain mode (spec_trigger) fires when the board reaches empty state (0 ready, 0 running). Recent fixes have addressed thin-goal task re-promotion loops and stale-in-review false positives, but three core patterns remain that block queue drain completion: thin-goal tasks stuck in rejection cycles, ambiguous task state transitions in unblock rules, and incomplete visibility into queue-drain trigger metrics.

## Goals

1. **Detect and prevent thin-goal task re-promotion cycles**: Implement a one-time rejection cache in board_worker that prevents the same task from bouncing between Blocked and Ready for AI when goal text remains <40 chars. Cache persists within a single board_worker cycle (per-role), resetting on startup.

2. **Harden STALE_IN_REVIEW rule to exclude open-PR tasks**: Refine board_unblock Rule 5 (STALE_IN_REVIEW) to check if the task has any open GitHub PR links (via PR URL pattern in description/comments) and skip stale-time enforcement for tasks with confirmed PRs, eliminating false-positive age-based demotions while PR review is active.

3. **Add queue-drain telemetry visibility**: Wire TriggerDetector counters (ready_count, running_count, thin_goal_blocked_count, stale_in_review_skip_count) into observability metrics so operators can track why queue drain is or isn't triggered, with per-role breakdowns for goal/test/improve workers.

4. **Document thin-goal task resolution workflow**: Create operator runbook at `docs/operator/thin_goal_resolution.md` that explains why tasks are blocked at <40 chars, how to unblock them (add detailed description, re-promote), and common patterns that trigger thin-goal rejection (single-word goals, copy-paste errors, incomplete spec payloads).

## Constraints

- Thin-goal rejection cache must be local to board_worker process; no persistent store needed.
- STALE_IN_REVIEW refinement must not make PR discovery calls to GitHub (best-effort pattern match only).
- Telemetry counters must use existing observability infrastructure (ExecutionTrace metadata or logs).
- Do not modify Plane board state or label policies; only improve decision logic around existing states.
- Spec-author tasks are exempt from thin-goal checks (they carry YAML payload, not goal text).

## Success Criteria

- Thin-goal tasks that were previously re-promoted within a single cycle are now rejected once and logged (no infinite loop).
- STALE_IN_REVIEW rule skips tasks with recognized PR URLs in description; test with at least 3 PR URL patterns (GitHub, GitLab, Gitea).
- TriggerDetector telemetry shows ready/running counts and reason code at each queue-drain check.
- Operator runbook is discoverable and explains workflow with examples.
- All changes have test coverage >=80% for new logic paths; existing tests remain passing.
