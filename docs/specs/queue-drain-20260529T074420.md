## Overview

The board_unblock rules (Rules 1–8) autonomously resolve queue stalls by moving tasks between Plane states based on label patterns and timeout thresholds. Recent improvements (ADR 0005, fix/board-worker, fix/taxonomy) surfaced three concrete queue drainage issues: false-positive stale-in-review on tasks with open PRs, thin-goal re-promotion loop cascades, and cross-lane executor classification ambiguity in task metadata. This campaign hardens queue state transitions and improves classification accuracy.

## Goals

1. **Fix stale-in-review false positives (Rule 5)** — Rule 5 moves tasks to Backlog when In Review >4h, but does not check whether the blocking PR is actually open. Integrate pr_review_watcher discovery or use minimal GitHub API probing to confirm PR open state before transitioning. Acceptance: Rule 5 test suite passes; zero false-positive In Review→Backlog transitions on real boards with open PRs in flight.

2. **Tighten thin-goal task classification** — board_worker creates thin-goal follow-ups from improve tasks; GOAL_BACKLOG_PROMOTE (Rule 7) re-promotes them when parent terminates. Recent thin-goal re-promotion loops (fix/board-worker PR) show Rule 7 overfires when task lineage is misclassified. Update board_worker and/or Rule 7 to enforce strict source labels (source: autonomy + source: improve-suggestion) and document the thin-goal parent→child lineage invariant. Acceptance: thin-goal re-promotion loop tests pass (from PR #170); board test suite green; documentation updated in board_worker/main.py.

3. **Align backend classification in task metadata** — Recent taxonomy fix (commit 155c8fc) re-classified aider_local and direct_local as direct worker backends, not executor lanes. Audit task metadata emitted by board_worker and ExecutionCoordinator to confirm backend_type labels match the taxonomy. Update any metadata serializers that still reference executor lanes. Acceptance: grep for executor-lane references in metadata returns zero; execution trace schema test validates backend names against CxRP 0.3.1 enum.

4. **Add queue depth metric to runbook** — Observability gap: board metrics (ready_count, running_count, blocked_count) are not surfaced in watchdog loop output. Add a simple queue-depth probe to operations-center-board-stats that emits ready/blocked/in-review/running counts per task-kind label. Acceptance: new operations-center-board-stats query; results logged to watchdog heartbeat; runbook section added to docs/operator/watchdog_loop.md.

## Constraints

- **No new backend enums** — backend classification uses existing CxRP 0.3.1 enum; do not propose new backend types.
- **Rule 5 must not require GitHub creds in read-only mode** — probe may be anonymous API call or must degrade gracefully if rate-limited.
- **Avoid re-opening ADR 0005 debate** — taxonomy is settled; focus on consistency enforcement.
- **Thin-goal lineage must be auditable** — every thin-goal task must have original-task-id: label; add detector to Custodian if absent.
